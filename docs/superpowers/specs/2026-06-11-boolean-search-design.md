# 불리언 검색 (OR / AND / 괄호) 설계

작성일: 2026-06-11

## 목표

검색을 단일 키워드에서 **불리언 표현식**으로 확장한다.

- `A|B|C` (OR) — 키워드 중 하나라도 포함한 행을 표시, 키워드별 **다른 색**.
- `A&B` (AND) — 같은 **행**에 모든 키워드가 포함된 경우만 표시, 키워드별 다른 색.
- `(A&B)|C` — 괄호로 OR/AND 조합.

적용 대상: **이 파일에서**(in-file 하이라이트)와 **같은 폴더/하위폴더/GIT**(cross-file 목록) 양쪽.

## 1. 공통 표현식 모델 (JS + Go 미러 구현)

프론트(in-file)와 백엔드(folder)가 동일하게 평가해야 하므로 같은 문법의 작은 재귀하강 파서를 **JS와 Go에 각각** 구현한다.

### 토큰화
- 연산자: `&` `|` `(` `)` 만.
- 그 외 텍스트는 키워드로 누적하고 양끝 공백을 trim → `hello world`는 공백 포함 한 구(句)로 검색.
- `"..."` 큰따옴표로 감싸면 내부의 `&|()`·공백은 리터럴로 취급(예: `"Tom & Jerry"`).
- 빈 키워드(연산자 사이 공백뿐)는 무시.

### 문법 / 우선순위
```
OR   := AND ('|' AND)*
AND  := atom ('&' atom)*
atom := '(' OR ')' | term
```
- **AND가 OR보다 먼저** 바인딩. 괄호로 재정의. `A&B|C` ≡ `(A&B)|C`.

### 평가
- 평가 단위는 **행(行)**. leaf = `행텍스트소문자.includes(키워드소문자)` (대소문자 무시, 기존과 동일).
- AND 노드 = 모든 자식 true, OR 노드 = 하나라도 true.

### 견고성 (fallback)
- 연산자가 전혀 없으면 입력 전체가 단일 키워드 → **기존 동작과 완전히 동일**(하위 호환).
- 괄호 불균형 등 파싱 실패 시 입력 전체를 리터럴 단일 키워드로 폴백 → 검색이 깨지지 않음.

### 색상 배정
- 표현식에 등장한 **서로 다른 키워드(소문자 기준)** 를 **첫 등장 순서**로 인덱싱.
- 8색 팔레트를 순환(`kw-0`…`kw-7`, 8개 초과 시 modulo). 같은 키워드는 항상 같은 색.
- 색상 맵은 프론트가 소유. 라이트/다크 모두 대비되도록 `color-mix`/`oklch` 기반 8색.

## 2. 이 파일에서 (in-file) — `highlightInFile` 개편

현재는 문서 전체 텍스트를 이어붙여 단일 키워드를 찾는다. AND("같은 행")를 위해 **행 단위 평가**로 전환한다.

- 텍스트 노드를 **가장 가까운 행 단위 블록**으로 그룹화하여 행텍스트를 구성:
  - 코드/표: `<tr>` (실제 소스 라인)
  - 마크다운 산문: `data-source-line`을 가진 블록(문단/제목/리스트 항목)이 한 "행"
  - 해소 불가 시 previewBody 직속 블록, 최종 fallback은 previewBody 자체
- 각 행에 표현식을 평가. **조건을 만족하는 행에서만** 각 키워드의 모든 등장을 해당 색 `<mark class="search-mark kw-N">`로 감싼다.
  - 단일 키워드면 모든 행이 자연스럽게 평가되어 기존과 동일하게 전부 하이라이트.
  - 순수 OR면 키워드를 가진 모든 행이 만족.
  - AND면 모든 키워드를 가진 행만.
- 인라인 요소를 가로지르는 매치는 **행 내부**에서 그대로 지원(현재 동작 유지). 행 경계를 넘는 매치는 찾지 않음(행 기반에서 자연스러움).
- 키워드 간 등장 구간이 겹치면 먼저 온(시작 위치 빠른) 것 우선, 겹치는 뒤 구간은 버림.
- 히트 객체에 `keyword`/색 인덱스 추가. 히트 리스트의 needle 스니펫 색상을 키워드 색으로 표시.
- `중요도/줄` 정렬·줄번호 해소(`lineNumberForHit`)·우선순위(`priorityForHit`)는 그대로 동작.
- 포커스(`.current`)는 배경을 accent로 덮지 않고 **외곽선/굵게**로 강조하여 키워드 색 정체성 유지.

### "행" 정의 주의
산문에서 "행"은 렌더된 블록(문단/제목/리스트 항목) 단위다. 같은 문단의 두 키워드는 AND 매치로 본다. 코드/CSV/표는 진짜 라인(`<tr>`) 단위다.

## 3. 같은 폴더 / 하위폴더 / GIT — 백엔드 `/api/search`

### 스코프 3단
- `이 폴더` (shallow): `state.folderSearchScope = ""`, `searchDirShallow(abs)` — 즉시 파일만.
- `하위폴더` (recursive): `scope=tree`, `searchTreeRecursive(abs, …)` — cwd 기준 재귀(git 무관).
- `GIT 전체`: `scope=git`, git root에서 재귀.

UI 토글은 `이 폴더 / 하위폴더 / GIT 전체` 3단.

### 문서 우선 필터 (코드 제외)
- 적용 범위: **`하위폴더` + `GIT 전체`만** (재귀 스코프). `이 폴더`(shallow)는 항상 전체 파일 검색.
- 기본값(쿼리 `allFiles` 미지정)일 때 **문서 확장자만** 스캔:
  `.md .markdown .mdx .txt .log`
  (CSV/TSV·코드/설정은 기본 제외 — 용량/노이즈 방지)
- `allFiles=1`이면 기존 텍스트 판별(`isProbablyText`) 전체로 확장(코드·CSV 포함).
- Go: `isDocExt(name)` 헬퍼 추가, `searchTreeRecursive`에 `docsOnly bool` 파라미터 추가(true면 비문서 파일 skip). `searchDirShallow`는 영향 없음.

### 파일별 평가
- 파일을 **줄 단위**로 읽어 각 줄에 표현식을 평가.
- 하나라도 만족하는 줄이 있으면 결과 포함. `count` = 조건 만족 줄 수.
- 파일에 등장한 키워드 집합을 `matchedTerms []string`로 응답에 추가.
- `searchResult`에 `MatchedTerms []string` 필드 추가(`json:"matchedTerms"`). 기존 `Snippets`는 유지.

### 결과 행 표시 (프론트)
- `파일명 + 키워드 색상 칩(A/B/C) + 카운트`. 칩 색은 프론트 색 맵으로 `matchedTerms`를 매핑.
- 결과 목록 **아래 "전체 파일 검색" 버튼**:
  - 노출 조건: 스코프 ∈ {tree, git} 이고 현재 요청이 문서 전용(allFiles 미설정)일 때.
  - 클릭 시 같은 질의/스코프를 `allFiles=1`로 재검색하고 버튼 숨김.
  - 질의·스코프 변경 시 `allFiles` 상태 초기화.

## 4. 상태/제약

- 프론트 전체가 `web.go`에 임베드 → **백틱 템플릿 리터럴 금지**, 문자열 결합 사용.
- i18n: 라벨 3곳 중복 목록 동기화. 신규: `하위폴더`/`Subfolder`, `전체 파일 검색`/`Search all files (incl. code)`.
- 신규 상태: `state.folderSearchAllFiles`(기본 false). `state.folderSearchScope`에 `"tree"` 추가.

## 5. 테스트

- Go 파서/평가 유닛 테스트: 토큰화(따옴표·공백구), 우선순위(`A&B|C`), 괄호(`(A&B)|C`), fallback(불균형 괄호 → 리터럴), 행 단위 AND.
- `searchTreeRecursive` docsOnly 필터 테스트(.go/.csv 제외, .md/.txt 포함).
- 기존 `search_test.go`는 줄 수=등장 수 케이스라 그대로 통과해야 함(회귀 가드).

## 하위 호환성

- 연산자·따옴표 없는 단일 질의는 기존과 동일 동작.
- 폴더 검색 `count`는 "등장 수"에서 "만족 줄 수"로 의미가 바뀌지만, 기존 테스트 데이터(한 줄당 한 등장)에서는 값이 동일.
- 리터럴 `&`/`|`를 검색하려면 `"..."`로 감싸야 함(드문 케이스, 알려진 변경점).
